Common Indexing Mistakes

One million product-catalog objects, five deliberately wrong schema choices. Run each card to see the failure mode. Fix it from the Weaviate console, then re-run to see the difference.

1. Range query with no rangeable index

Range filters on price_cents have no rangeable inverted index and no filterable index, so Weaviate refuses the query outright with requires inverted index. From the catalog operator's POV, the storefront just broke.

price_cents int filterable=false rangeable=false 

{
  Get {
    IndexingMistakesDemo(
      where: {
        operator: And,
        operands: [
          { path: ["price_cents"], operator: GreaterThanEqual, valueInt: 30000 },
          { path: ["price_cents"], operator: LessThanEqual,    valueInt: 40000 }
        ]
      },
      limit: 10
    ) {
      name brand category price_cents sku
    }
  }
}

2. File path searched with WORD tokenization

BM25 search for /products/cameras/gopro against a WORD-tokenized path field. The slashes and dots are dropped, so the term matches any path containing the literal word gopro anywhere - for example in unrelated categories.

spec_sheet_path text tokenization=word 

{
  Get {
    IndexingMistakesDemo(
      bm25: { query: "/products/cameras/gopro", properties: ["spec_sheet_path"] },
      limit: 10
    ) {
      spec_sheet_path category brand name
    }
  }
}

3. SKU searched with WORD tokenization

BM25 lookup for GP-ACT-BLK-001 against a WORD-tokenized SKU field. The hyphens split the SKU into four tokens, so the lookup matches every product sharing any of those tokens.

sku text tokenization=word 

{
  Get {
    IndexingMistakesDemo(
      bm25: { query: "GP-ACT-BLK-001", properties: ["sku"] },
      limit: 10
    ) {
      sku brand category name
    }
  }
}

4. Equality filter on a non-filterable property

Filter category == "Cameras > Action" against a property with no inverted filterable bucket. Same fate as the price card: Weaviate refuses with requires inverted index. The category facet on the storefront stops working.

category text filterable=false tokenization=field 

{
  Get {
    IndexingMistakesDemo(
      where: { path: ["category"], operator: Equal, valueText: "Cameras > Action" },
      limit: 10
    ) {
      name brand category sku price_cents
    }
  }
}

5. Email searched with WORD tokenization

BM25 search for info@nikon.com against a WORD-tokenized email field. The query splits on @ and . into info, nikon, com. No product actually uses info@ as a prefix, so the top results all say support@nikon.com - BM25 matched on the domain tokens and silently returned the wrong addresses.

support_email text tokenization=word 

{
  Get {
    IndexingMistakesDemo(
      bm25: { query: "info@nikon.com", properties: ["support_email"] },
      limit: 10
    ) {
      support_email brand category name
    }
  }
}

Press Run query to execute.
